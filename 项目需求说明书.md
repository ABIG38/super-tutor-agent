## 1. 项目背景与目标

### 1.1 背景痛点

1. **信息过载**：学生需翻阅大量教材、真题、考纲，难以高效整合
2. **规划盲目**：缺乏基于真实教材内容的个性化复习计划生成工具
3. **AI 幻觉**：通用大模型回答专业课时常捏造知识点，且不标注信息来源
4. **上下文受限**：多数产品无法同时处理整本教材（数十万字）

### 1.2 项目目标

构建一款 **无幻觉、带溯源** 的学业/考研规划桌面应用，核心能力：

1. 支持上传多本教材，进行跨章节全局检索
2. 所有回答强制标注信息来源，杜绝捏造
3. 基于教材真实内容自动生成按周/天拆解的复习计划
4. 专业课术语同时支持语义搜索和关键词精确匹配

> **⚠️ 阶段范围**
>
> |      | 阶段一（本项目）                                | 阶段二（后续）                                        |
> | ---- | ----------------------------------------------- | ----------------------------------------------------- |
> | 范围 | 上传→索引→提问→溯源回答；按需生成一次性复习计划 | 多轮对话、进度反馈闭环、动态计划调整、Context Caching |
> | 定位 | 被动问答 + 一次性规划                           | 主动辅导 + 持续跟踪                                   |

## 2. 用户角色

| 角色       | 描述                  | 核心需求                                  |
| ---------- | --------------------- | ----------------------------------------- |
| 考研学生   | 备考研究生入学考试    | 上传教材/真题，做知识点问答，获取复习计划 |
| 学业规划师 | 辅导机构老师/学长学姐 | 基于教材批量生成规划，审核溯源内容        |
| 教务管理者 | 院系/机构管理人员     | 查看使用情况，管理公共知识库              |

**阶段一仅覆盖「考研学生」。**

### 2.5 主界面布局概述

阶段一主界面为 PySide6 桌面窗口，整体分为三个功能区域：

| 区域           | 位置             | 内容                                                                           |
| -------------- | ---------------- | ------------------------------------------------------------------------------ |
| **知识库列表** | 左侧面板         | 已上传文档列表（显示文件名、文档类型标记、chunk 数）。支持右键删除/重命名      |
| **问答 Tab**   | 中央主区域       | 聊天气泡式对话界面，用户输入框在底部，回答显示来源标注。右上角「清空对话」按钮 |
| **规划 Tab**   | 中央主区域切换页 | 参数设置表单（天数/学时/起始章节）+ 生成按钮 + 生成结果 Markdown 预览          |

底部状态栏实时显示当前操作进度（如「正在索引…」「检索到 12 个片段…」）。

## 3. 功能需求

### 3.1 文档管理

| 编号 | 功能           | 描述                                                                                                                                                                                                                                                     | 优先级 |
| ---- | -------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------ |
| F-01 | 文档上传       | 支持 PDF、Word(.docx)、Markdown、TXT。不支持旧版 .doc。单个文件 ≤ 200MB。支持从文件对话框选择和从系统桌面拖拽文件到窗口两种方式                                                                                                                          | P0     |
| F-02 | 文档解析       | 提取文本，保留原始段落结构（连续文本块，由空行或缩进分界）和 Markdown 标题层级。扫描版 PDF（无文字层）解析后提示用户「该文档为扫描件，未提取到文字」。加密 PDF 提示输入密码，错误则拒绝。数学公式保留 LaTeX 原文；图片/图表不解析                        | P0     |
| F-03 | 文档管理       | 查看已上传文档列表（显示文件名、文档类型标记）、删除（弹窗二次确认后删除索引和原始文件）、修改应用内显示名称（不改动磁盘原始文件）。文档类型支持标记为「教材」或「真题」（标记影响计划生成范围）。重复上传同一文件时提示「已存在」，让用户选择跳过或覆盖 | P0     |
| F-20 | 模型检测与引导 | 首次启动时检测本地 Embedding / Reranker 模型文件是否存在（默认路径 `knowledge_base/models/`）。若缺失，弹窗引导用户自动下载或提示手动放置；下载失败时给出清晰指引。若模型文件损坏，提示重新下载。避免用户首次使用时因模型缺失报不明错误                  | P0     |
| F-04 | 批量上传       | 一次上传多本教材或多份真题                                                                                                                                                                                                                               | P1     |
| F-05 | 文档预览       | 应用内查看已上传文档的解析后文本                                                                                                                                                                                                                         | P2     |

### 3.2 智能问答

| 编号 | 功能       | 描述                                                                                                                                                                                                                                                 | 优先级 |
| ---- | ---------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------ |
| F-06 | 知识点问答 | 基于已上传文档回答专业课问题。教材和真题均参与检索；仅标记为「教材」的文档参与规划生成。若知识库为空（未上传任何文档），提示用户先上传文档。用户不满意的回答可重新生成（重新检索+LLM）。支持清空当前对话                                             | P0     |
| F-07 | 溯源引用   | 每条回答必须标注 `[来源文档名]`。若 chunk 元数据中包含章节字段则标注 `[来源文档名：章节]`；若 chunk 不包含章节字段，只标文档名，禁止 LLM 自行编造章节号。来源必须与检索结果一一对应，不可张冠李戴                                                    | P0     |
| F-08 | 无数据声明 | 当上传文档无法回答时，必须明确告知用户。触发条件：① 检索最高分 < 0.3（代码层拦截）；② LLM 判断 <context> 内容不含实质答案时（LLM 自省）。两种情况均回复「未在上传文档中找到相关答案，请检查文档内容或更换提问方式」，禁止 LLM 自行调用通用知识库补充 | P0     |
| F-09 | 混合检索   | 同时使用语义检索和关键词检索，结果融合去重后传给 LLM。中文教材为主场景，英文查询不在阶段一优化范围内                                                                                                                                                 | P0     |
| F-10 | 上下文缓存 | 利用 LLM API 缓存加速重复查询（阶段二实现，阶段一仅留配置入口）                                                                                                                                                                                      | P2     |
| F-11 | 多轮对话   | 支持基于历史上下文的连续追问（阶段二实现）                                                                                                                                                                                                           | P2     |

### 3.3 复习规划

| 编号 | 功能     | 描述                                                                                                                                                                                                                                                                                                                                                                                                                | 优先级 |
| ---- | -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------ |
| F-12 | 计划生成 | 基于教材章节目录和重难点，生成按周/天拆解的 Markdown 计划。参数：天数（必填，正整数）+ 每日学时（选填，正整数）+ 起始章节（选填，支持「第X章」「第X节」「章节名关键词」，系统自动匹配最近似的章节）。若知识库中无标记为「教材」的文档 → 提示「请先上传含目录的教材」。若教材存在但检测不到章节目录 → 降级处理：按知识点生成计划，并告知用户「未检测到标准目录，以下按知识点生成」。用户不满意计划可调整参数重新生成 | P0     |
| F-13 | 计划导出 | 将计划导出为 Markdown / PDF 文件                                                                                                                                                                                                                                                                                                                                                                                    | P1     |
| F-14 | 计划调整 | 根据用户实际进度动态调整后续计划（依赖阶段二）                                                                                                                                                                                                                                                                                                                                                                      | P2     |
| F-15 | 进度追踪 | 标记已掌握/未掌握知识点，可视化进度（依赖阶段二）                                                                                                                                                                                                                                                                                                                                                                   | P2     |

> F-14/F-15 阶段一仅预留接口。需要用户反馈数据作为输入，属于阶段二的「学习反馈闭环」。

| 优先级       | 数量 | 功能编号                         |

## 4. 非功能需求

### 4.1 性能

| 编号  | 指标         | 目标值                                                                                                                                                                                                                                     |
| ----- | ------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| NF-01 | 文档处理速度 | 100 页 PDF：解析+切分+向量化+BM25 ≤ 60 秒（基准硬件：i5/8GB RAM/SSD）。单个文件 ≤ 200MB，超出则拒绝并提示                                                                                                                                  |
| NF-02 | 问答响应时间 | 首次问答（含检索+LLM）≤ 8 秒（同上硬件基准）。LLM API 超时重试 1 次，仍失败则提示用户「网络异常，请稍后重试」                                                                                                                              |
| NF-03 | 检索上下文量 | 单次传给 LLM 的检索结果 ≤ 8000 字符                                                                                                                                                                                                        |
| NF-04 | 桌面流畅度   | 单用户运行不卡 UI；上传索引、问答、规划均在后台线程异步执行，每个阶段通过状态栏文字实时反馈当前进度（如「📖 正在解析文档…」「🔍 正在检索…」）。索引过程中用户关闭应用，下次启动自动恢复（ChromaDB 自带持久化，BM25 pickle 按事务边界保存） |

### 4.2 准确性

| 编号  | 指标           | 目标值                                                                                                                                   |
| ----- | -------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| NF-05 | 溯源准确率     | ≥ 95% 回答包含正确来源标注。从预设测试问题集（≥50 条）中随机抽检                                                                         |
| NF-06 | 幻觉率         | ≤ 5%。定义：① 回答声称引用自上传文档、但该内容在文档中实际查不到；② LLM 自行编造的章节号、页码等元数据也视为幻觉。基于测试问题集人工评估 |
| NF-07 | 术语检索命中率 | BM25 对专业术语命中 ≥ 90%。取 30 个专业术语，逐条验证 Top-3 是否命中                                                                     |

### 4.3 可用性

| 编号  | 指标       | 目标值                                                                                                                                                        |
| ----- | ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| NF-08 | 启动可靠性 | 启动成功率 ≥ 99%，不闪退。首次启动时若 .env 不存在或 API Key 为空，弹窗引导用户配置而非崩溃                                                                   |
| NF-09 | 上手时间   | 新用户 5 分钟内完成首次上传+问答                                                                                                                              |
| NF-10 | 磁盘空间   | 索引前检查磁盘剩余空间 ≥ 索引文件预估大小的 2 倍，不足则提示用户清理。本地数据存储根目录默认 `knowledge_base/`，支持用户在设置中自定义路径（防 C 盘空间不足） |

### 4.4 安全与隐私

| 编号  | 要求         | 说明                                                                                                                     |
| ----- | ------------ | ------------------------------------------------------------------------------------------------------------------------ |
| NF-11 | 数据本地化   | 向量库、索引、原始文档均存储于本机文件系统，不上传云端                                                                   |
| NF-12 | 删除完整性   | 用户在应用中删除文档时，同步清理向量和 BM25 索引，不留残留。历史聊天中引用已删除文档的旧回答，来源标注保留但可能无法复核 |
| NF-13 | API Key 保护 | 通过 .env 文件读取，不硬编码、不写入日志、不出现在错误提示中。.env 缺失时引导用户创建，不崩溃                            |
| NF-14 | 版本升级兼容 | 阶段一至阶段二的版本升级，旧版 ChromaDB 数据和索引文件可直接加载使用，不做格式迁移                                       |

### 4.5 运行环境

| 编号  | 指标     | 目标值                                                                                                                                        |
| ----- | -------- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| NF-19 | 操作系统 | 阶段一仅支持 Windows 10/11 64-bit。macOS / Linux 作为阶段二可选扩展                                                                           |
| NF-20 | 最低配置 | CPU: 4 核 2.5GHz, RAM: 8GB, 磁盘: 10GB 可用空间, 显示器: 1280×720                                                                             |
| NF-21 | 推荐配置 | CPU: 6 核 3.0GHz, RAM: 16GB, 磁盘: 20GB+ 可用空间 (SSD), 显示器: 1920×1080                                                                    |

## 7. 术语表

| 术语            | 说明                                                                                       |
| --------------- | ------------------------------------------------------------------------------------------ |
| RAG             | Retrieval-Augmented Generation — 检索增强生成：先从知识库检索相关内容，再交给 LLM 生成回答 |
| BM25            | 基于词频-逆文档频率的关键词检索算法，适合专业术语精确匹配                                  |
| Context Caching | LLM API 的上下文缓存：固定内容只计算一次，后续请求复用，降低延迟和费用                     |
| Embedding       | 将文本转换为数学向量的过程，向量之间的距离代表语义相似度                                   |
| chunk           | 文档切分后的最小检索单元，每块约 800 字符                                                  |
| 溯源/引用       | 回答中标明信息来自哪本教材、哪个章节                                                       |
