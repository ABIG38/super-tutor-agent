## 1. 项目背景与目标

### 1.1 背景痛点

1. **信息过载**：学生需翻阅大量教材、真题、考纲，难以高效整合
2. **规划盲目**：缺乏基于真实教材内容的个性化复习计划生成工具
3. **AI 幻觉**：通用大模型回答专业课时常捏造知识点，且不标注信息来源
4. **上下文受限**：多数产品无法同时处理整本教材（数十万字）

### 1.2 项目目标

构建一款 **无幻觉、带溯源** 的学业/考研规划桌面应用，核心能力：

1. 支持上传多本教材，进行跨章节全局检索
2. 所有回答强制标注信息来源，杜绝捏造
3. 基于教材真实内容自动生成按周/天拆解的复习计划
4. 专业课术语同时支持语义搜索和关键词精确匹配
5. 支持多课程（科目）独立管理，数据相互隔离

### 3.3 智能问答

| 编号 | 功能               | 描述                                                                                                                                                                                                                                                                                                                                 | 优先级 |
| ---- | ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------ |
| F-06 | 知识点问答         | 基于已上传文档回答专业课问题。教材和真题均参与检索；仅标记为「教材」的文档参与规划生成。若知识库为空（未上传任何文档），提示用户先上传文档。用户不满意回答可重新生成（重新检索+LLM）。支持清空当前对话                                                                                                                               | P0     |
| F-07 | 溯源引用           | 每条回答必须标注 `[来源文档名]`。若 chunk 元数据中包含章节字段则标注 `[来源文档名：章节]`；若 chunk 不包含章节字段，只标文档名，禁止 LLM 自行编造章节号。来源必须与检索结果一一对应，不可张冠李戴                                                                                                                                    | P0     |
| F-08 | 无数据声明         | 当上传文档无法回答时，必须明确告知用户。触发条件：① 检索最高分 < 0.3（代码层拦截）；② LLM 判断 `<context>` 内容不含实质答案时（LLM 自省）。两种情况均回复「未在上传文档中找到相关答案，请检查文档内容或更换提问方式」，禁止 LLM 自行调用通用知识库补充                                                                                   | P0     |
| F-24 | 意图路由           | 用户输入经意图路由模块处理：规则前置短路（日常寒暄直通 chat；含「计划+生成/制定/帮我」关键词的直通 plan）→ LLM 分类（rag/chat/plan）。chat 意图不触发任何检索，plan 意图引流至规划 Tab，rag 意图走完整 RAG 流程。避免用户闲聊时浪费 API 费用和检索时间                                                                               | P0     |
| F-09 | 混合检索           | 同时使用语义检索（向量 Top-10）和关键词检索（BM25 Top-10），经 RRF（倒数秩融合）算法融合去重后，再经 Cross-Encoder Reranker 精排至 Top-5 传给 LLM。中文教材为主场景                                                                                                                                                                     | P0     |
| F-25 | 会话管理           | 支持创建多个对话会话，左侧列表展示。每个会话独立保存历史记录（JSONL 格式）。支持会话重命名、删除、清空。启动时自动恢复上一次会话。会话数据持久化到 `knowledge_base/index/chats/{id}.jsonl` + `manifest.json`                                                                                                                          | P0     |
| F-26 | 流式渲染           | LLM 回答令牌逐次推送到前端，QWebEngineView 实时渲染 Markdown（含表格、代码块、标题层级）。支持 `<think>` 推理过程折叠/展开显示（DeepSeek Reasoning 格式）。每 10 个 token 刷新一次渲染，避免频繁渲染卡顿                                                                                                                              | P0     |
| F-16 | 查询重写           | 检索前将用户口语化/短提问改写为独立的搜索关键词短语。触发条件：对话历史非空且用户提问 ≤ 15 字。LLM 结合最近 4 轮对话消解指代（如"它的时间复杂度呢"→ 重写为"快速排序的时间复杂度"）。提升 BM25 和向量检索命中率                                                                                                                     | **P0** |
| F-28 | 对话历史压缩       | 累计超过 8 轮对话时，自动将旧历史通过 LLM 总结为一段摘要（保留核心意图、提及的概念和关键结论），替换为一条压缩后的 system 消息。避免超出上下文窗口，同时保留连续追问所需的背景信息                                                                                      | **P1** |
| F-29 | 计划注入问答上下文 | 当存在活跃计划时，将当日任务内容自动注入到 RAG 上下文中。用户提问时 LLM 能感知「用户今天第 X 天，任务是什么」，回答可结合计划进度做个性化建议（如「根据您今天的计划，建议先复习…」）                                                                                  | **P1** |

## 4. 非功能需求

### 4.1 性能

| 编号  | 指标           | 目标值                                                                                                                                                                                                                                         |
| ----- | -------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| NF-01 | 文档处理速度   | 100 页 PDF：解析+切分+向量化+BM25 ≤ 60 秒（基准硬件：i5/8GB RAM/SSD）。单个文件 ≤ 200MB，超出则拒绝并提示                                                                                                                                        |
| NF-02 | 问答响应时间   | 首次问答（含检索+LLM）≤ 8 秒（同上硬件基准）。LLM API 超时重试 1 次，仍失败则提示用户「网络异常，请稍后重试」                                                                                                                                    |
| NF-03 | 检索上下文量   | 单次传给 LLM 的检索结果 ≤ 8000 字符                                                                                                                                                                                                              |
| NF-04 | 桌面流畅度     | 单用户运行不卡 UI；上传索引、问答、计划均在后台线程异步执行，每个阶段通过状态栏文字实时反馈当前进度（如「📖 正在解析文档…」「🔍 正在检索…」）。索引过程中用户关闭应用，下次启动自动恢复（ChromaDB 自带持久化，BM25 pickle 按事务边界保存）       |
| NF-23 | 模型延迟加载   | Embedding 模型在首次检索或写入时按需加载，应用启动时不加载（避免启动卡死）。Reranker 模型按需加载，加载失败降级为原序                                                                                                                             |
| NF-24 | 流式渲染频率   | LLM 流式输出每 10 个 token 刷新一次前端渲染，避免每 token 刷新导致的界面卡顿                                                                                                                                                                      |

### 4.2 准确性

| 编号  | 指标           | 目标值                                                                                                                                       |
| ----- | -------------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| NF-05 | 溯源准确率     | ≥ 95% 回答包含正确来源标注。从预设测试问题集（≥50 条）中随机抽检                                                                             |
| NF-06 | 幻觉率         | ≤ 5%。定义：① 回答声称引用自上传文档、但该内容在文档中实际查不到；② LLM 自行编造的章节号、页码等元数据也视为幻觉。基于测试问题集人工评估 |
| NF-07 | 术语检索命中率 | BM25 对专业术语命中 ≥ 90%。取 30 个专业术语，逐条验证 Top-3 是否命中                                                                         |

### 4.3 可用性

| 编号  | 指标         | 目标值                                                                                                                                                        |
| ----- | ------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| NF-08 | 启动可靠性   | 启动成功率 ≥ 99%，不闪退。首次启动时若 .env 不存在或 API Key 为空，弹窗引导用户配置而非崩溃                                                                   |
| NF-09 | 上手时间     | 新用户 5 分钟内完成首次上传+问答                                                                                                                              |
| NF-10 | 磁盘空间     | 索引前检查磁盘剩余空间 ≥ 索引文件预估大小的 2 倍，不足则提示用户清理。本地数据存储根目录默认 `knowledge_base/`，支持用户在设置中自定义路径（防 C 盘空间不足） |
| NF-25 | 优雅降级     | Reranker 加载失败 → 自动降级为 RRF 排序结果，不阻塞问答流程。网络搜索超时 → 忽略搜索，仅基于本地知识库回答。Embedding 模型加载失败 → 提示用户检查模型文件   |

### 4.4 安全与隐私

| 编号  | 要求         | 说明                                                                                                                                         |
| ----- | ------------ | -------------------------------------------------------------------------------------------------------------------------------------------- |
| NF-11 | 数据本地化   | 向量库、索引、原始文档均存储于本机文件系统，不上传云端                                                                                       |
| NF-12 | 删除完整性   | 用户在应用中删除文档时，同步清理向量和 BM25 索引，不留残留。删除课程时同步清除该课程所有文档和进度数据                                      |
| NF-13 | API Key 保护 | 通过 `.env` 文件读取，不硬编码、不写入日志、不在错误提示中明文显示。设置页面使用密码模式输入框（`EchoMode.Password`）                        |

## 5. 风险评估

| 风险                      | 影响 | 概率 | 应对                                                       |
| ------------------------- | ---- | ---- | ---------------------------------------------------------- |
| LLM API 费用超预期        | 中   | 中   | 检索上下文控制在 8000 字符以内，后续可引入 Context Caching   |
| PDF 公式/图表解析质量差   | 高   | 中   | 公式保留 LaTeX 原文；图表阶段一跳过                        |
| 检索召回率不足            | 高   | 低   | 混合检索（向量+BM25+RRF 融合）+ Reranker 精排互补         |
| 用户隐私顾虑              | 中   | 低   | 全部数据存本地，隐私政策按需出具                           |
| 360 搜索接口不稳定        | 低   | 中   | 4 秒超时自动降级，不阻塞本地 RAG 流程                    |
| Embedding 模型加载失败    | 高   | 低   | 启动时检测模型完整性；加载失败提示用户检查模型文件路径     |
| LLM 意图路由误判          | 中   | 低   | 规则前置短路覆盖高频词；LLM 分类失败降级为 RAG，确保兜底   |

## 6. 术语表

| 术语              | 说明                                                                                             |
| ----------------- | ------------------------------------------------------------------------------------------------ |
| RAG               | Retrieval-Augmented Generation — 检索增强生成：先从知识库检索相关内容，再交给 LLM 生成回答       |
| BM25              | 基于词频-逆文档频率的关键词检索算法，适合专业术语精确匹配                                        |
| RRF               | Reciprocal Rank Fusion — 倒数秩融合：将多个检索结果按排名倒数加权融合，无需调参               |
| Reranker          | Cross-Encoder 重排序模型：对检索结果做精细化的语义相关性打分，提升 Top-K 准确率                 |
| Context Caching   | LLM API 的上下文缓存：固定内容只计算一次，后续请求复用，降低延迟和费用                           |
| Embedding         | 将文本转换为数学向量的过程，向量之间的距离代表语义相似度                                         |
| chunk             | 文档切分后的最小检索单元，每块约 800 字符                                                        |
| 溯源/引用         | 回答中标明信息来自哪本教材、哪个章节                                                             |
| Intent Routing    | 意图路由：将用户输入分类为 rag（知识问答）/ chat（闲聊）/ plan（规划），决定处理流程             |
| Query Rewriting   | 查询重写：基于对话历史将短提问/代词消解为独立完整的搜索词，提升检索命中率                       |
| History Compression | 历史压缩：超过 8 轮对话时自动总结旧历史为摘要，避免超出 LLM 上下文窗口                         |
| Reasoning Block   | 思维链块：DeepSeek 等模型的 `<think>...</think>` 推理过程展示，前端支持折叠展开                 |
| Course Isolation  | 课程隔离：每门课程拥有独立的文档、索引和计划数据，课程间数据互不干扰                             |
