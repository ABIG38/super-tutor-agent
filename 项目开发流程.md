## 1. 技术可行性评估

### 1.1 可行方案

| 模块 | 方案 | 可行性 | 风险 |
|------|------|--------|------|
| PDF 解析 | PyMuPDF (fitz) | ✅ 已验证 | 扫描件/加密已处理，公式保留 LaTeX |
| DOCX 解析 | python-docx | ✅ 已验证 | 仅支持 .docx |
| 文本切分 | RecursiveCharacterTextSplitter + tiktoken | ✅ 成熟方案 | 需选择与 LLM 匹配的 tokenizer |
| 向量存储 | ChromaDB 持久化 | ✅ 已验证 | 本地运行，无外部依赖 |
| Embedding | BAAI/bge-small-zh-v1.5 | ✅ 已验证 | 需下载模型文件（~100MB），CPU 可运行 |
| BM25 检索 | rank-bm25 + jieba | ✅ 成熟方案 | 纯内存，需 pickle 持久化 |
| Reranker | BAAI/bge-reranker-base | ✅ 已验证 | 需下载模型（~1.1GB），CPU 推理 1-2 秒 |
| LLM 调用 | OpenAI 兼容 API (DeepSeek) | ✅ 已验证 | 依赖网络，需 API Key |
| 桌面 UI | PySide6 | ✅ 已验证 | 无边框窗口、暗色主题已实现 |
| 进度存储 | SQLite (WAL 模式) | ✅ 已验证 | 本地文件，并发安全 |
| 配置管理 | Pydantic BaseSettings | ✅ 成熟方案 | 自动校验，类型安全 |
| 日志 | loguru 按天滚动 | ✅ 成熟方案 | 零配置，脱敏 filter |
| 打包 | PyInstaller | ✅ 已验证 | 需处理模型文件外置 |

### 1.2 风险与缓解

| 风险 | 等级 | 缓解措施 |
|------|------|----------|
| BM25 + 向量混合检索的 RRF 参数需要调优 | 低 | RRF k=60 是论文推荐值，稳定可用 |
| Reranker 模型下载失败 | 中 | F-20 引导下载流程；打包时内置模型 |
| 中文分词 jieba 对专业术语效果 | 低 | 混合检索互补；阶段二加查询重写 |
| ChromaDB 数据量增长后检索变慢 | 低 | 单用户场景，教材量有限（< 100 本） |
| Docling 解析复杂 PDF 表格失败 | 中 | Fallback 到 PyMuPDF 纯文本；关闭 OCR 避免性能问题 |
| PyInstaller 打包后模型路径问题 | 中 | .spec 文件管理，HF_HUB_OFFLINE=1 |

### 任务 A3: 升级文档切分器

## 4. 阶段一-B：检索增强

> **目标**: 实现混合检索 + Reranker，使问答质量达到 NF-05~NF-07 标准。

### 任务 B1: 实现 BM25 搜索引擎
- **文件**: `backend/retrieval/bm25_search.py` (新建)
- **内容**:
  - `BM25Searcher` 类
  - `build_index(corpus)` — 全量重建
  - `search(query, top_k)` — 关键词检索
  - 中文用 `jieba` 分词，英文用空格
  - pickle 持久化：「临时文件 + 原子 rename」防断电
  - 删除时维护 `_deleted_source_ids`，检索时过滤
  - 启动自动从 pickle 加载或重建
- **依赖**: A1
- **验收**: 上传教材后，`bm25.search("时间复杂度")` 返回相关 chunks 且命中率 ≥ 90%

### 任务 B2: 实现混合检索融合
- **文件**: `backend/retrieval/hybrid_search.py` (新建)
- **内容**:
  - `HybridSearcher` 类，封装 VectorStore + BM25Searcher
  - `search(query, top_k)` 方法：
    1. VectorStore.search(top_k=5) + BM25Searcher.search(top_k=5)
    2. RRF 融合：`score = Σ 1/(60 + rank)`
    3. 按 RRF 分数降序取 Top 10-15
    4. 阈值过滤：最高分 < 0.3 → 返回空
    5. MD5 去重
  - `search_raw(query, top_k)` — 不过滤（供 planner 使用）
- **依赖**: A5, B1
- **验收**: 同一查询在混合模式下命中率高于纯向量/纯 BM25

### 任务 B3: 实现 Reranker 精排
- **文件**: `backend/retrieval/reranker.py` (新建)
- **内容**:
  - `Reranker` 类，使用 `BAAI/bge-reranker-base`
  - 延迟加载（首次调用时加载模型）
  - `rerank(query, chunks)` → 返回带 Cross-Encoder 分数的排序列表
  - 分数过滤：低于阈值的丢弃
  - Device 自动检测：CUDA 可用则 GPU，否则 CPU
  - 集成到 HybridSearcher：RRF Top 15 → Reranker → Top 7
- **依赖**: B2, A4
- **验收**: 引入 Reranker 后，假阳性 chunk 减少 ≥ 30%

### 任务 B4: 更新 Orchestrator 使用混合检索
- **文件**: `backend/agent/orchestrator.py` (修改)
- **内容**:
  - 替换 `self.vector_store` 为 `self.hybrid_search = HybridSearcher()`
  - `ask()` 方法使用 `hybrid_search.search()` 替代 `vector_store.search()`
  - 空结果时返回友好提示
  - `generate_plan()` 使用 `hybrid_search.search_raw()`
- **依赖**: B3
- **验收**: 问答流程端到端走混合检索 + Reranker

## 5. 阶段一-C：前后端对接

> **目标**: 将前后端完全对接，实现完整的用户交互闭环。

### 任务 C1: 文档上传 → 索引流程对接
- **涉及文件**: `document_tree.py`, `orchestrator.py`
- **内容**:
  - `DocumentTree._upload_document()` → 触发 `SuperTutorAgent.ingest_document()`
  - 在后台线程（`QThread`）执行索引，避免阻塞 UI
  - 通过信号更新状态栏：`"📖 正在解析 {filename}..."` → `"🔍 正在索引 (12/45)..."` → `"✅ 索引完成"`
  - 扫描件/加密/过大文件的错误弹窗
  - 重复文件检测 → 弹窗"跳过/覆盖"
  - 索引完成后刷新文档树
- **依赖**: B4
- **验收**: 拖拽 PDF 到窗口 → 自动索引 → 文档树显示文件名 → 状态栏显示进度

### 任务 C2: 问答流程对接（流式渲染）
- **涉及文件**: `chat_page.py`, `orchestrator.py`
- **内容**:
  - ChatPage 发送按钮 → `SuperTutorAgent.ask(query)` 流式调用
  - 在后台线程运行检索+LLM，通过信号逐 token 发送到主线程
  - `chat_page.add_user_message()` + 流式 `add_assistant_token()` + `add_assistant_done()`
  - 停止按钮 → `llm.cancel_stream()`
  - 知识库为空时的提示
  - 检索无结果时的提示
  - API 错误的 toast 提示
- **依赖**: B4
- **验收**: 输入"什么是时间复杂度？" → 流式显示回答 → 末尾标注来源

### 任务 C3: 规划流程对接
- **涉及文件**: `plan_page.py`, `orchestrator.py`, `tracker.py`
- **内容**:
  - PlanPage 生成按钮 → `SuperTutorAgent.generate_plan(days, hours)`
  - 异步执行（后台线程）
  - 返回后渲染每日任务卡片（`add_day_card`）
  - 复选框勾选 → `tracker.mark_task(task_id, completed)`
  - 重新生成按钮 → 跳过已学章节
  - 导出按钮 → 保存为 Markdown 文件
  - 无教材时提示
- **依赖**: B4
- **验收**: 设置 30 天/2 小时 → 生成 → 显示每日卡片 → 打勾 → 进度条更新 → 重启后进度保留

### 任务 C4: 状态栏实时反馈
- **涉及文件**: `desktop_app.py`, 各 page
- **内容**:
  - 全局状态栏信号：`status_update(text, color)`
  - 索引中：绿色圆点 + "正在索引数据结构.pdf..."
  - 检索中：绿色圆点 + "检索到 12 个相关片段..."
  - LLM 生成中：绿色圆点 + "正在生成回答..."
  - 空闲：灰色圆点 + "系统就绪"
  - 错误：红色圆点 + 错误信息
- **依赖**: C1, C2, C3
- **验收**: 每个操作都在状态栏有对应的文字反馈

## 6. 阶段一-D：边界情况与测试

> **目标**: 补齐技术设计文档 §9 的所有边界情况处理，并通过测试验证。

### 任务 D1: 补齐 Orchestrator 边界处理
- **文件**: `backend/agent/orchestrator.py` (修改)
- **内容对照技术设计 §9**:

| # | 情况 | 实现 |
|---|------|------|
| ④ | 重复上传同名文件 | 检查 `_sources` 集合，返回 `duplicate` → UI 弹窗 |
| ⑤ | 知识库为空时提问 | 检查 sources 为空 → 不调检索 → 直接返回提示 |
| ⑥ | 检索分数过低 | `merged[0]["RRF_score"] < 0.3` → 返回空 → 提示"未找到" |
| ⑩ | 索引中途关闭 | ChromaDB 自动持久化 + BM25 pickle 原子写入（已实现） |
| ⑭ | 磁盘空间不足 | 索引前 `shutil.disk_usage()` → < 2x 预估 → raise OSError |
| ⑰ | 索引异常回滚 | `try/except` 包裹 → 清理 ChromaDB 本次 IDs → BM25 重建 |
| ⑳ | 文档重命名 | `rename_document(old, new)` → 更新 metadata + 重建 BM25 |

- **依赖**: B4
- **验收**: 逐条对照测试，每种边界情况触发正确响应
